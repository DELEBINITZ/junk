Executive summary

FortiRecon Assistant is a conversational AI platform that adapts its identity and abilities to whatever capabilities we enable. A master orchestrator coordinates a team of named specialist agents — Atlas (product guidance) today, Sentinel (security reports) and an external-attack-surface agent next — and answers the user in one consistent voice. Today only Atlas is enabled, so the assistant presents as a product guide; the same system, unchanged, becomes a security analyst the moment further specialists are switched on.

The platform is organised in three tiers: a stateless application service that holds the agent logic, a model-serving tier (language, embedding, and re-ranking models on GPUs) shared across the fleet, and a data tier (a vector database and PostgreSQL). This separation is the basis of its scalability — the application scales horizontally, while the expensive model tier scales independently and is used economically through a three-lane routing scheme that sends most work to a small, cheap model.

Two design decisions define the product. First, the assistant’s persona, scope, and boundaries are derived automatically from its enabled capabilities — one platform serves many products without a rebuild. Second, it never cold-rejects a valid request or leaves a user at a dead-end. A prior version rejected legitimate questions as “out of scope”; the architecture now makes that structurally impossible — anything it cannot do is met with a graceful, guiding response. This document covers the whole system: its components, a request’s journey through them, how the platform scales and deploys, and its security and operational posture.

Overview & design principles

FortiRecon Assistant follows the supervisor / orchestrator-workers pattern, a well-established design for reliable multi-agent systems. A single master orchestrator stays in conversation with the user and delegates bounded sub-tasks to a team of named specialist agents — Atlas, Sentinel, and others. The specialists never address the user directly and never call one another; the orchestrator alone decides what runs, coordinates it, and composes the final answer in one consistent voice (specialist names are internal — the user hears one assistant).

Control flow is deterministic, expressed as a directed graph (LangGraph). This is deliberate: a language model decides what to do at each decision point, but the graph decides where control can go. That separation makes the system debuggable, testable, and safe to operate — behaviour is auditable rather than emergent.

Why it matters

Deterministic orchestration turns “an AI that usually does the right thing” into a system with guarantees: every request passes the same safety gate, every answer passes the same privacy filter, and every failure has a defined fallback.

System architecture & components

The platform is a set of independently-scalable tiers. The application service is stateless — it keeps no conversation in memory; all durable state lives in the data tier — which is what lets it run as many identical replicas behind a load balancer. The model-serving and data tiers are shared services that every replica calls.

ComponentRole
Application serviceFastAPI process hosting the API, authentication, the orchestrator graph, and guardrails. Fully async; no in-memory session state.
LLM (three lanes)An OpenAI-compatible model server (e.g. Qwen on vLLM/SGLang). A small fast model handles routing, planning, and summarising; a large standard model composes answers; a deep lane handles the hardest multi-step analysis.
Embeddings & re-rankerText-Embeddings-Inference (TEI) services turn text into vectors and re-score retrieved passages. Ingestion and query use the same embedding model, so they always agree.
QdrantVector database holding the searchable knowledge. Every query is filtered to the requesting organisation.
PostgreSQLTwo roles: the conversation store (chat history + rolling summaries) and durable orchestrator state (LangGraph checkpoints), each behind an async connection pool.
MCP serversExternal capabilities connected over the Model Context Protocol; discovered and wired in at start-up from configuration.
Ingestion serviceA standalone, offline job that keeps the knowledge base current. Never on the live request path.
3

The request lifecycle

A single question travels a well-defined path across the tiers. Understanding this journey is the clearest way to see how the pieces fit.

  1. Authenticate. The request carries a gateway API key and a signed identity token (JWT). The service verifies both and derives the caller’s organisation, user, and roles. Identity is established before anything else runs.
  2. Screen & classify — in parallel. Two fast-lane model calls run concurrently: an attack screen and an intent classification. Running them together removes a round-trip of latency.
  3. Route. The results join at a routing decision: answer directly, use one capability, plan across several, gracefully clarify, or — only for a genuine attack — stop.
  4. Retrieve & reason. Chosen capabilities run (in parallel where independent), retrieving org-scoped passages from the vector store and reasoning over them.
  5. Reflect. If nothing useful came back, the orchestrator re-plans once or degrades gracefully — it never dead-ends.
  6. Compose & stream. The standard-lane model composes one grounded, cited answer, streamed token-by-token to the user.
  7. Guard & persist. The answer is redacted for personal data and checked for any instruction leak, then the turn is saved and the conversation summary updated for future context.
Latency by design

Most requests never pay for a planning step (the router writes the task inline), the two pre-routing calls overlap, and independent capabilities run concurrently — so the common case is fast, and only genuinely complex questions incur the full pipeline.

4

Agent orchestration

The orchestrator is the deterministic graph at the heart of the application tier. Every message follows the same structure; all three answer paths converge on one privacy filter before reaching the user.

Sub-agents run in one of two modes: a single-call mode for pure retrieval (one search, no loop — cheapest), or a reasoning mode where the agent chains tools, for example searching and then fetching a full documentation page so a step-by-step walkthrough arrives complete. The product-guide capability runs in reasoning mode.

5

Derived identity & capabilities

An assistant’s personality is normally written into its prompts. Ours is not. At start-up the system inspects which capabilities registered and builds a single SystemProfile — a name, a one-line role, a description of what it can help with, and the domains it covers. Every persona-bearing instruction (the router, conversational replies, the answer composer, the security classifier) is a template filled from that profile.

The consequence is direct: change the enabled capabilities and the whole personality re-forms — name, scope, and boundaries — with no change to code or prompts.

The master and its named specialists

The user-facing assistant is the master (“FortiRecon Assistant”). Under it sits a team of named specialist agents — Atlas answers product-guide questions from the documentation, Sentinel answers from the security-reports corpus, and further agents (e.g. external attack surface) plug in the same way. These names are internal: the master coordinates them and answers in one voice, so a user always hears a single assistant, not a hand-off between bots. Enable more specialists and the master’s advertised scope grows; its identity is derived from whichever are switched on.

Request routing

Routing is cost-aware and, by design, incapable of rejecting a request for being off-topic. A fast classifier assigns every message one of four intents; only a genuine attack — handled separately — can stop a turn.

IntentWhenHandling
DirectAnswerable without a capability — greeting, “who are you”, thanks.A brief conversational reply.
SimpleOne capability plausibly fits.Task written inline; the planner is skipped.
ComplexNeeds several capabilities or cross-referencing.A planner decomposes it into a task graph.
ClarifyNothing clearly fits, or the request is too vague.A warm capability redirect — not a refusal.

Safety and scope are separate concerns

  • Safety — prompt injection, jailbreak, and extraction attempts are the only hard stops, isolated in the security gate.
  • Scope — never a rejection. The router has no “refuse” action; anything unclear becomes a guiding redirect.
  • A deterministic safeguard guarantees a non-empty system can never dead-end: with one capability enabled, an unclear question is attempted against it; with several, it becomes a clarify redirect.

The reliability improvement

Previously

Cold rejection

A hardcoded refusal path and a fixed security persona judged valid product questions “out of scope.”

“How do I export my findings?”
→ “That’s outside what I can help with.”
Now

Attempt, then guide

The question is tried against the enabled capability; any miss degrades to a helpful redirect.

“How do I export my findings?”
→ searches the guide → answers, or offers what it can do.

Retrieval & knowledge

The assistant answers from source material, not from memory. A specialist retrieves relevant passages from the vector store and reasons over them, fetching a full page when a walkthrough needs the complete section — so answers arrive whole, not as truncated fragments.

  • Domain-aware enrichment — a query is expanded along the right axis for its corpus; a documentation question is not reformulated as if it were a threat query.
  • Fusion & re-ranking — results from several query variants are merged (Reciprocal Rank Fusion) and re-scored by a cross-encoder, lifting the most relevant passages to the top.
  • Grounded citation — passages carry their title, section heading, and navigation path, so the assistant can tell a user exactly where to go in the product. Internal relevance scores never leave the retrieval layer, and no internal identifiers reach the user’s answer — a page reference is passed to the model internally only so it can fetch a full page when a walkthrough needs one.
8

Knowledge ingestion

Answer quality depends on the knowledge base being correct and current. A dedicated ingestion service converts the product documentation into embedded, searchable passages — using the same embedding model the application queries with, so the two always agree. Its defining property: it is safe to re-run at any time, and after every run the database is an exact mirror of the source.

GuaranteeHow it is achieved
No duplicatesEach page and chunk has a stable, unique identifier, so re-ingesting overwrites in place rather than creating a second copy.
No collisionsIdentifiers use the full document path, so similarly-named pages in different sections never overwrite one another.
No stale contentWhen a page is shortened, its leftover fragments are pruned; edited content never lingers.
Removed pages disappearAfter a full run, pages deleted or renamed in the source are removed, so restructured docs never surface outdated answers.
Net effect

On any re-run: new pages created, changed pages overwritten, removed pages deleted — with zero duplicates and zero stale content. The job is guarded so a failed run can never wipe the corpus.

Scalability & performance

The architecture is built so that load can be met by adding hardware to the right tier, not by re-engineering. Four properties carry the scaling story.

1 · A stateless application tier scales horizontally

The application service keeps no conversation in memory — every durable fact lives in PostgreSQL (history and orchestrator checkpoints) and Qdrant. Any replica can serve any request, so capacity for more concurrent chats is simply more identical replicas behind a load balancer. Because the service is fully asynchronous, a single replica handles many in-flight requests while they wait on model and database calls.

2 · Three model lanes make inference affordable

Model inference is the dominant cost and the main throughput constraint. The platform addresses this directly by routing each step to the cheapest model that can do it.

LaneModel classUsed for
FastSmall (≈7B)Routing, planning, summarising — the high-volume, low-difficulty steps.
StandardLarge (≈72B)Composing the final grounded answer.
DeepStrongestThe hardest multi-step analysis — opt-in, and quota-capped.

The effect is that the expensive large model is reserved for answer composition, while the frequent bookkeeping steps run on a small, cheap model. Lanes scale as separate model deployments, so each is sized to its own demand.

3 · Work is parallelised within a request

  • The two pre-routing model calls (attack screen and intent) run concurrently.
  • Independent capabilities dispatch in parallel batches, so a multi-part question is answered in the time of its slowest part, not the sum.
  • Roughly four in five requests skip the planning step entirely — the router writes the capability’s task inline.

4 · Backends scale independently

Qdrant (sharded, replicated, with vector quantisation to cut memory) and PostgreSQL (connection-pooled, with a high-availability configuration for production) each scale on their own axis, decoupled from the application fleet and from each other. Knowledge ingestion runs offline, so refreshing the corpus never competes with live traffic.

Under load

Timeouts, fail-open safety, and defined fallbacks at every node mean a traffic spike degrades gracefully rather than failing. Admission control (bounded concurrency returning a clean “busy” signal instead of a brownout) and per-tenant fairness quotas are the recommended next hardening step for high scale.

5 · Cost & capacity

Cost is dominated by GPU model serving, and the three-lane scheme is the primary lever against it — the majority of calls run on the small model, and the large model is reserved for composing answers. Latency follows the same shape: a simple question returns quickly (no planning step), while a genuinely complex, multi-capability question incurs the full pipeline. Formal capacity, latency, and throughput benchmarks against production hardware are a defined pre-launch activity; they size each model lane to a measured demand model rather than an estimate.

Deployment topology

The application is a standard container with no special runtime requirements, which keeps deployment cloud-agnostic and the operating model simple. It runs at three scales.

ScaleShapeWhen
Single boxOne host running the app plus its backends via containers (Docker Compose).Development, demos, small pilots.
Managed servicesThe container behind a load balancer; managed database and vector store; GPU model endpoints.Production for a single product like the current Atlas.
KubernetesApp as an auto-scaling Deployment (≥2 replicas); model lanes on GPU node pools scaled on queue depth; Qdrant and PostgreSQL as highly-available stateful sets.Multi-tenant, high-volume operation.

The dividing line is deliberate: the stateless application tier scales elastically (add replicas), while the stateful and GPU tiers are provisioned to a measured capacity model. Today Atlas ships as a single product deployment with one capability enabled; broadening it is configuration, and scaling it is provisioning — neither is a redesign.

Availability & durability. In the scaled configurations the stateful tiers run highly available — Qdrant sharded and replicated, PostgreSQL in a primary/replica arrangement — with regular backups of both the conversation/state database and the vector store, so the loss of a node degrades service rather than interrupting it, and the knowledge base can be rebuilt from source by re-running ingestion (§8).

Security, privacy & isolation

Defence is layered and, importantly, aware of the assistant’s own identity, so it protects this deployment rather than a fixed persona.

Authentication

Every request carries two credentials: a gateway API key and a signed identity token (JWT). The service verifies both and reads the organisation, user, and roles from the token’s claims. In production, missing or invalid credentials are rejected outright.

Attack defence (input)

A fast pattern layer plus an always-on classifier detect attempts to manipulate the assistant — overriding its instructions, extracting its configuration, or jailbreaking it. The classifier judges attacks on the assistant only; an unusual or off-topic but harmless question is never treated as a threat.

If the classifier step is slow to respond, the system prioritises availability and lets the request proceed rather than blocking the user — a deliberate trade-off. The residual exposure is limited: the fast pattern layer is not subject to that timeout and still runs, and the output filter (below) always runs regardless. The enterprise hardening on the roadmap tightens this further.

Privacy (output)

Every answer — from a capability, a conversational reply, or a redirect — passes a filter that removes personal data (PII: names, contact, financial and health identifiers) and blocks any leak of the system’s own instructions. Redaction also applies to the live token stream, so nothing sensitive appears even momentarily.

Data residency

The language, embedding, and re-ranking models are self-hosted on our own infrastructure (OpenAI-compatible — e.g. Qwen served on vLLM/SGLang). Customer questions and retrieved content are processed inside our perimeter and are not sent to any third-party AI provider — a deliberate choice for a security product handling sensitive data. A managed external model is a supported configuration option, but is not used for sensitive deployments.

External capabilities

Connecting a capability served by an external tool server (over MCP) introduces a trust boundary: the relevant query context may leave our perimeter to reach it. Such servers are vetted before enablement, calls carry a short-lived, organisation-scoped credential, and any state-changing external action is designed to require explicit human approval. The capability shipping today (the product guide) is in-perimeter retrieval only, so no such boundary is crossed in the current deployment.

Tenant isolation

Isolation is enforced by explicit organisation scoping on every query: identity establishes the organisation, and every vector search and database read is filtered to it — a tenant can only see content marked public or tagged to it, and deleted content is never returned. Database-enforced row-level security is the recommended additional layer for production hardening.

Note on current vs. hardened state

The controls above are implemented today. Enterprise-grade additions on the roadmap — database row-level security, external identity (OIDC), rate limiting, and human approval gates for any state-changing action — are provisioned for but not all wired in the current single-capability deployment.

Extensibility & roadmap

Adding a capability requires no changes to the orchestrator, router, planner, or persona. A capability registers itself, and everything downstream — routing, planning, the advertised personality, the metadata endpoint, and the user interface — adapts automatically.

  • An in-house capability is a small module plus one registration line.
  • An external capability served over the Model Context Protocol is configuration only: declare its address and description, and it is discovered and wired in at start-up.
# adding a third-party “Brand Protection” capability — no code change
MCP_SERVERS={"brand": {"url":"https://brand-mcp/mcp",
  "display_name":"Brand Protection",
  "description":"Monitors brand abuse, phishing, typosquatting."}}

Roadmap

  • Near term — enable the Sentinel (reports) and external-attack-surface specialists behind the same FortiRecon Assistant front end.
  • Answer-quality assurance — an automated “judge” pass that critiques answer quality, not merely whether content was found.
  • Enterprise hardening — database row-level security, external identity (OIDC), rate limiting, and approval gates for state-changing actions.
  • Routing at scale — the current design serves roughly fifteen to twenty capabilities on one router; beyond that, an embedding-based capability index takes over.

Operational readiness

ConcernProvision
ReliabilityEvery model-calling step has a timeout, and every node has a defined fallback — the system degrades gracefully rather than hanging or failing.
ResponsivenessAnswers stream token-by-token; most requests skip the planning step; independent work runs in parallel.
ContinuityMulti-turn memory via a rolling summary plus a recent-message window; orchestrator state is checkpointed durably in PostgreSQL, so long runs are resumable.
ObservabilityStructured tracing across the pipeline; a metadata endpoint exposes the live identity for any client.
VerificationAn automated routing test-suite asserts which capabilities run for representative queries, guarding against regressions.
Known limits — stated plainly

Routing on a single prompt scales to roughly twenty capabilities; the reflection loop recovers from empty results but does not yet judge answer quality. Both are on the roadmap and neither affects the current single-capability deployment.

Appendix — glossary & stack

Glossary

Orchestrator / supervisor
The single agent that talks to the user and coordinates the specialist capabilities.
Capability (sub-agent)
A specialist that performs a bounded task — e.g. searching the product documentation.
Stateless service
An application process that stores no session in memory, so any copy can serve any request — the basis of horizontal scaling.
Lane
A model tier chosen for a step by cost/difficulty — fast, standard, or deep.
Derived identity
The assistant’s name, scope, and boundaries, generated at start-up from the enabled capabilities.
RAG (retrieval-augmented generation)
Answering from retrieved source passages rather than the model’s memory, so answers are grounded and citable.
Guardrail
An automated safety/privacy check on inputs and outputs.
PII
Personally identifiable information — names, contact details, and financial or health identifiers that must be redacted.
MCP (Model Context Protocol)
An open standard for connecting external tools/capabilities to an assistant.
Checkpoint
A durable snapshot of orchestrator state, so a run can resume after interruption.
