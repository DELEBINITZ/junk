==============================================================================
 GPU MODEL DEPLOYMENT — Agentic Security Intelligence platform
==============================================================================
Every model the app needs, the EXACT server/endpoint it expects, launch commands,
ports, VRAM sizing, and health checks. Ports + hostnames match infra/.env.production.

The app calls each model over HTTP. It does NOT load any model in-process EXCEPT
PII (Presidio + spaCy, CPU — see §6). So this file = 3 model servers + 3 infra
services. NO guard-model servers: injection + content safety run on the MAIN LLM
(LLMJudgeGuard). Get the ENDPOINT SHAPES right and the app just works.

------------------------------------------------------------------------------
 0. INVENTORY  (what runs where)
------------------------------------------------------------------------------
  SERVICE        MODEL                              SERVER   PORT   APP ENV VAR
  -----------    --------------------------------   ------   ----   ----------------
  vllm           Qwen/Qwen2.5-72B-Instruct (prod)   vLLM     8000   VLLM_BASE_URL
                 Qwen/Qwen2.5-32B-Instruct (staging) — ONE model per env; it also
                 doubles as the security judge (LLMJudgeGuard — no guard servers)
  tei-embed      Qwen/Qwen3-Embedding-4B            TEI      8080   TEI_EMBED_URL
  tei-rerank     BAAI/bge-reranker-v2-m3 *          TEI      8081   TEI_RERANK_URL
  -----------    PII: Presidio + en_core_web_lg     in-app   —      (CPU, §6)
  infra          Qdrant / Postgres / Redis          —        6333/5432/6379

  * prod intent is "Qwen3-Reranker"; bge-reranker-v2-m3 is the SAFE TEI choice —
    see §3 for the Qwen3-Reranker caveat.

  GATED MODELS: NONE — every model above is ungated. HF_TOKEN not needed.

 1. VRAM SIZING & LAYOUT  (fp16 unless noted)
------------------------------------------------------------------------------
  MODEL                      WEIGHTS   + KV/runtime   MIN GPUs
  Qwen2.5-72B-Instruct       ~145 GB   +20-40 GB      2x80GB (TP2) | 4x48GB (TP4)
    (FP8/AWQ ~75 GB)                                  1x80GB tight | 2x48GB
  Qwen2.5-32B-Instruct       ~65 GB    +10-20 GB      1x80GB | 2x48GB (staging)
    (AWQ ~18 GB)                                      1x24GB tight
  Qwen3-Embedding-4B         ~8 GB     small          1x24GB  (8B=~16GB, dim 4096)
  bge-reranker-v2-m3         ~1.1 GB   small          shares a GPU

  RECOMMENDED LAYOUTS
    A) One 8xH100/A100-80GB box (most headroom)
         GPU0-1  72B  (TP2)
         GPU2    tei-embed + tei-rerank
    B) Budget 2x80GB only
         72B (TP2) fills both -> run embed/rerank on a 3rd 24GB aux card,
         or AWQ the 72B to free room. Do NOT starve KV cache (the LLM also
         serves the guardrail-judge calls).
    C) 4x48GB (A6000/L40S)
         72B-AWQ (TP4) | embed+rerank on a spare card or CPU-adjacent box

  Pin GPUs per container with:  --gpus '"device=2"'   (or CUDA_VISIBLE_DEVICES)

  TEI IMAGE — simplest: the universal CUDA tag (works across recent arches; what
  staging runs):
    ghcr.io/huggingface/text-embeddings-inference:cuda-1.9
  Older per-arch 1.6 tags also work if you need them:
    H100 hopper-1.6 | A100 1.6 | A10/A6000/3090 86-1.6 | L4/L40S/4090 89-1.6 | T4 turing-1.6
  Below this file uses <TEI_IMAGE> — substitute your tag.
  NOTE on pooling: TEI 1.9 auto-detects Qwen3-Embedding's last-token pooling from
  the model repo (1_Pooling/config.json), so --pooling last-token is optional there;
  keeping the flag explicit is harmless and survives older images. VERIFY in logs:
  `docker logs tei-embed 2>&1 | grep -i pool` should say LastToken.

==============================================================================
 2. LLM — ONE model on vLLM  (port 8000; 72B in prod, 32B in staging)
==============================================================================
App posts OpenAI-style to VLLM_BASE_URL + /chat/completions. ONE served model per
environment — all lanes (fast/standard/deep) collapse to it, and it ALSO serves
the guardrail judge (LLMJudgeGuard) — so this single server is the only LLM.
--served-model-name MUST equal the app's VLLM_MODEL exactly.

  # prod (72B):
  docker run -d --name vllm --gpus all --shm-size 32g \
    -p 8000:8000 \
    -v ~/.cache/huggingface:/root/.cache/huggingface \
    vllm/vllm-openai:latest \
    --model Qwen/Qwen2.5-72B-Instruct \
    --served-model-name Qwen/Qwen2.5-72B-Instruct \
    --tensor-parallel-size 2 \
    --max-model-len 32768

  # staging (32B): same command with
  #   --model Qwen/Qwen2.5-32B-Instruct --served-model-name Qwen/Qwen2.5-32B-Instruct
  # (and VLLM_MODEL=Qwen/Qwen2.5-32B-Instruct in the app env)

  Quantize to fit smaller VRAM: add  --quantization fp8  (or use an -AWQ checkpoint
  with --quantization awq). Size KV cache generously: every agent step AND every
  guardrail-judge call go through this one server.

==============================================================================
 3. RERANKER — TEI cross-encoder  (port 8081)
==============================================================================
App posts:  POST {TEI_RERANK_URL}/rerank   {"query": "...", "texts": ["...","..."]}
So the served model MUST be a CROSS-ENCODER / sequence-classification reranker that
TEI exposes on /rerank. bge-reranker-v2-m3 fits exactly.

  docker run -d --name tei-rerank --gpus '"device=4"' \
    -p 8081:80 \
    -v ~/.cache/huggingface:/data \
    -e HF_TOKEN=$HF_TOKEN \
    <TEI_IMAGE> \
    --model-id BAAI/bge-reranker-v2-m3

  QWEN3-RERANKER CAVEAT: prod comments say "Qwen3-Reranker". Qwen3-Reranker is an
  LLM-style (yes/no) reranker — NOT a classic cross-encoder, and stock TEI /rerank
  may not serve it. If your TEI build supports it, swap --model-id to
  Qwen/Qwen3-Reranker-0.6B (or -4B). Otherwise stay on bge-reranker-v2-m3 (strong,
  multilingual, TEI-native). Reranking has no dim constraint, so this is free to swap.

==============================================================================
 4. EMBEDDER — Qwen3-Embedding-4B on TEI  (port 8080)   *** READ THE DIM NOTE ***
==============================================================================
App posts:  POST {TEI_EMBED_URL}/embed   {"inputs": ["...","..."]}  -> [[float,...], ...]
Qwen3-Embedding uses LAST-TOKEN pooling.

  docker run -d --name tei-embed --gpus '"device=2"' \
    -p 8080:80 \
    -v ~/.cache/huggingface:/data \
    -e HF_TOKEN=$HF_TOKEN \
    <TEI_IMAGE> \
    --model-id Qwen/Qwen3-Embedding-4B \
    --pooling last-token \
    --max-client-batch-size 64

  *** EMBEDDING_DIM — ONE NUMBER, EVERYWHERE ***
  Qwen3-Embedding-4B native output = 2560 (the 24GB quant profile; the 8B is 4096).
  These FOUR must be identical or retrieval breaks (Qdrant dim error / 0 hits):
     1) what TEI returns          (4B native 2560)
     2) app  EMBEDDING_DIM=2560   (app/config.py + .env.production)
     3) cron EMBEDDING_DIM=2560   (scripts/index_reports_cron.py env)
     4) the Qdrant 'reports_kb' collection, created at 2560
  STANDARDIZE ON 2560. Switching embedder model (e.g. back to 8B@4096 or to 0.6B@1024)
  means: change all four AND drop+recreate the collection AND re-index every report.

  *** QUERY INSTRUCTION ***
  TEI does NOT auto-add Qwen3's query instruction. Set it on the APP (query side only;
  the cron embeds passages raw):
     EMBEDDING_QUERY_INSTRUCTION="Instruct: Given a web search query, retrieve relevant passages that answer the query\nQuery: "

==============================================================================
 5. GUARDRAILS — NO servers to deploy
==============================================================================
Injection + content safety run ON THE MAIN LLM (§2): the input pipeline sends one
small, hardened judge completion (LLMJudgeGuard — fenced untrusted input, strict
JSON verdict, temperature 0, max_tokens 96) per turn and BLOCKS on injection/
unsafe. Fail policy: GUARDRAILS_FAIL_CLOSED=true blocks when the judge errors.
Nothing to serve, nothing to configure beyond the LLM itself.
(Secret redaction, indirect-injection neutralization, output exfiltration defang
run in-process; PII is Presidio in-process — see §6.)

==============================================================================
 6. PII — Presidio + spaCy  (IN-APP, CPU — no server)
==============================================================================
Runs inside the app process; just install the extra + the spaCy model where the app runs:
     pip install -e ".[pii]"
     python -m spacy download en_core_web_lg
Security-tuned PII_ENTITIES deliberately omits IP_ADDRESS/URL/DOMAIN — they are
the product's subject matter, not PII.

 8. HEALTH CHECKS — exact shapes the app uses (run after each server is up)
==============================================================================
  # LLM (also serves the guardrail judge; staging: model = Qwen/Qwen2.5-32B-Instruct)
  curl -s localhost:8000/v1/chat/completions -H 'content-type: application/json' \
    -d '{"model":"Qwen/Qwen2.5-72B-Instruct","messages":[{"role":"user","content":"ping"}],"max_tokens":8}'

  # Embedder — vector length MUST be 2560 (Qwen3-Embedding-4B)
  curl -s localhost:8080/embed -H 'content-type: application/json' \
    -d '{"inputs":["hello world"]}' | python -c "import sys,json;print('dim',len(json.load(sys.stdin)[0]))"

  # Reranker
  curl -s localhost:8081/rerank -H 'content-type: application/json' \
    -d '{"query":"open ports","texts":["exposed port 22","cooking recipe"]}'

 9. WIRE THE APP  (infra/.env.production — already templated; confirm these)
==============================================================================
  LLM_PROVIDER=vllm
  VLLM_BASE_URL=http://<host>:8000/v1
  VLLM_MODEL=Qwen/Qwen2.5-72B-Instruct        # staging: Qwen/Qwen2.5-32B-Instruct
  EMBEDDING_PROVIDER=tei      TEI_EMBED_URL=http://<host>:8080      EMBEDDING_DIM=2560
  EMBEDDING_MODEL=Qwen/Qwen3-Embedding-4B
  EMBEDDING_QUERY_INSTRUCTION="Instruct: Given a web search query, retrieve relevant passages that answer the query\nQuery: "
  RERANK_ENABLED=true   RERANK_PROVIDER=tei   TEI_RERANK_URL=http://<host>:8081
  QDRANT_URL=http://<host>:6333
  # guardrails: nothing to wire — the judge runs on VLLM_BASE_URL automatically.

  CRON (scripts/index_reports_cron.py) must use the SAME embedder + dim:
  TEI_EMBED_URL=http://<host>:8080   EMBEDDING_DIM=2560   COLLECTION_NAME=reports_kb

  ORDER: bring up infra + all 3 model servers, pass §8 health checks, THEN start the
  app. In prod the boot guard refuses to start if a required setting is missing.
